📊 Resumen Ejecutivo

El Sistema Gestor Dental es una aplicación web moderna desarrollada para la gestión integral de clínicas dentales. El sistema proporciona herramientas completas para la administración de pacientes, citas, historiales médicos y odontogramas, sin incluir funcionalidades de inteligencia artificial.

  6. Respuesta: Datos devueltos al frontend

🗄️ Base de Datos

Arquitectura de Base de Datos

El sistema utiliza Supabase como plataforma de base de datos, que proporciona PostgreSQL con funcionalidades adicionales como autenticación, almacenamiento de archivos y APIs en tiempo real.

Esquema de Tablas

Tabla Propósito Campos Principales Relaciones
profiles Perfiles de usuario id, full_name, role, created_at auth.users (1:1)
patients Datos de pacientes id, name, age, email, phone, medical_history profiles (N:1), appointments (1:N)
appointments Citas médicas id, patient_id, date, time, status, notes patients (N:1)
consultations Consultas médicas id, patient_id, date, diagnosis, treatment patients (N:1), treatments (1:N)
odontograms Odontogramas digitales id, patient_id, chart_data, created_at patients (N:1)
files Archivos del paciente id, patient_id, file_name, file_path, file_type patients (N:1)
medical_histories Historiales médicos id, patient_id, data (JSONB) patients (N:1)
consultation_treatments Tratamientos por consulta id, consultation_id, treatment_type, cost consultations (N:1)

Políticas de Seguridad (RLS)

-- Ejemplo de políticas RLS implementadas CREATE POLICY "mis pacientes: leer" ON public.patients FOR SELECT USING (auth.uid() = dentist_id); CREATE POLICY "citas por dueño" ON public.appointments FOR ALL USING (EXISTS ( SELECT 1 FROM public.patients p WHERE p.id = patient_id AND p.dentist_id = auth.uid() ));

Índices y Optimización

  • 🔍 Índices Primarios: UUID para todas las tablas principales
  • ⚡ Índices Secundarios: En campos de búsqueda frecuente
  • 🔄 Timestamps: created_at y updated_at para auditoría
  • 📊 JSONB: Para datos flexibles como historiales médicos

🔐 Sistema de Autenticación

Arquitectura de Autenticación

🔑 Flujo de Autenticación

1. Usuario ingresa credenciales
2. Supabase Auth valida credenciales
3. Se genera JWT token
4. Token se almacena en localStorage
5. RouteGuard verifica token en cada ruta
6. Hook useRoleRedirect determina destino
7. Usuario accede a funcionalidades según rol

Roles y Permisos

Rol Acceso Funcionalidades Restricciones
Dentista Completo Gestión de pacientes, citas, odontogramas, archivos Solo sus pacientes
Paciente Limitado Ver perfil, citas propias, chat básico Solo su información

Componentes de Seguridad

  • 🛡️ RouteGuard: Componente que protege rutas según rol
  • 🔄 useRoleRedirect: Hook para redirección automática
  • 🔐 useSupabase: Hook para gestión de autenticación
  • ⏰ Expiración de Tokens: Renovación automática
  • 🚪 Logout Seguro: Limpieza de sesión completa

🔌 API y Endpoints

Arquitectura de API

El sistema utiliza una arquitectura híbrida que combina:

  • 🔗 Supabase Client: Para operaciones CRUD directas
  • 🖥️ Express.js Server: Para lógica de negocio compleja
  • 🔐 Autenticación: Integrada con Supabase Auth

Endpoints del Backend

🛡️ Seguridad y Privacidad

Medidas de Seguridad Implementadas

  • 🔐 Autenticación JWT: Tokens seguros con Supabase Auth
  • 🛡️ Row Level Security (RLS): Control granular de acceso a datos
  • 🔒 CORS Configurado: Restricción de dominios permitidos
  • 📝 Validación de Entrada: Sanitización de datos de usuario
  • ⏰ Expiración de Sesiones: Tokens con tiempo de vida limitado
  • 🔍 Auditoría: Logs de acceso y modificaciones

Políticas de Privacidad

  • 📊 Datos Personales: Encriptación en tránsito y reposo
  • 👥 Separación de Roles: Pacientes solo ven su información
  • 🗑️ Eliminación Segura: Borrado completo de datos sensibles
  • 📋 Consentimiento: Manejo de permisos de datos médicos

Compliance y Regulaciones

🔒 Cumplimiento HIPAA: El sistema está diseñado para cumplir con las regulaciones de privacidad médica, incluyendo encriptación de datos, control de acceso y auditoría de registros.
